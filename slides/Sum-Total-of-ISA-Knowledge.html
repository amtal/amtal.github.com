Types of Failures

  1. False positives discovering more false positives (sev: high)
  2. Underapproximations makes you re-visit code (sev: annoying)
  3. Script stomped over manually-entered markup (sev: only happens once)
  • at scale/academic view:
    • under/overapproximation of states,
    • true/false positives/negatives (and recall/precision from those)
  • in manual use, two failures you care about:
    • automated garbage cannon
    • you thought the script you ran was total but it weren't

Program Analysis is a Search Problem

  • Fast backtracking vs slow complex search
  • Specialized algorithms vs generic solver
  • Heuristics compensating for generic solver
  • Checking results of ∃∃ search vs ∀∀ search
  • Approximating state coverage via path coverage
  • this is as tl;dr as I can get a publication/tool review:
  • all program analysis is search over program state transitions
  • choice of fast stochastic search (fuzzer) vs transform-then-search (symbolic exec)
  • heuristics tend to be tied to a specific problem
    • modern sym. ex. tries to keep SMT solver very generic
    • code wrapping solver has the really problem-specific heuristics
  • this is important when reasoning about correctness, since:
    • existance of a path matching some constraints is easy to verify against HW
    • non-existance of a path requires trust in toolchain...
    • toolchains for slow-but-fancy analysis are huge
  • equivalence checking = proof of non-existance of differing outputs
    • exhaustive search is great if tractable
      • need a ton of ISA knowledge to select portions where it is
    • block/path coverage is an approximation of exhaustive search
      • only tracks $rip portion of program state
    • symbolic execution's "unsat" answer is exhaustive search on an SMT-internal IR
      • as trustworthy as 300k+ lines of C++ can be
      • plus our own transforms prior to SMT solver
  • in short: symbolic execution research doesn't offer a free lunch :(

What Went Right &

What Went Wrong

  • remember the original goal - bringing up decent static RE tools should be a small part of the total workflow
  • if these techniques race against someone just reading the manual and manually writing lifts for anything interesting
    • who wins?
    • under what conditions?
  1. Approximations:
    • acceptable, but validate major assumptions
  2. Partial lifting:
    • acceptable and commonplace
  3. Emulator-as-oracle:
    • less partial, needs a map to lifted model
  4. Full equivalence checking versus emulator:
    • hampered by 2 and 3, but sometimes works
  • good news:
    • querying a reference emulator is occasionally helpful compared to reading docs
    • trivial equivalence checking via symbolic execution is very occasionally useful
      • e.g. quickly checking 32-bit helper functions on 8051
    • lifting only some instructions and approximating a lot turns out to be fine for manual RE!
  • bad news:
    • you can't automate away deciding what to approximate, or mapping your architecture to your analysis tool
    • in a race against someone /w a ref. manual and basic skills, tools are only a minor advantage
    • you won't reach anywhere near academic levels of "success" (>50% instructions proven equivalent) on hacker tools
      • but you don't need to
